# DocuMind — Grounded Q&A for the Laravel Documentation

DocuMind answers plain-English questions about Laravel by retrieving the
relevant parts of Laravel's official documentation and having an LLM answer
**only from those retrieved docs** — with citations — rather than from the
model's own memory. It's a Retrieval-Augmented Generation (RAG) system, not a
generic chatbot: the grounding is the point.

**🔗 Live demo:** https://laravel-docs-assistant.streamlit.app/

> I decided to build this project to further deepen my knowledge about rag system and architecture through practical implementation. Why I chose this project ? That's because I have experience working with laravel in an ecommerce company. So I could actually verify if the answer generated by the llm is factual from the documents I embedded or is the llm model just hallucinating.

## Why grounding matters (the core idea)

A plain LLM answers from memory and will confidently invent things. Asked
"how do I make a single-action controller?", ungrounded Gemini returned
`php artisan make:controller --invoke` — a flag **that does not exist**. The
real flag is `--invokable`. DocuMind retrieves the actual `controllers.md`
section and answers from it, returning `--invokable` verbatim with the source.
Same model, same question — the difference is grounding.

## How it works (the RAG pipeline)

1. **Load** — read Laravel's Markdown docs into LangChain `Document` objects,
   tagging each with its source filename (for citations). Meta/navigation files
   (`documentation.md`, `readme.md`, `license.md`) are excluded so they don't
   pollute retrieval.
2. **Split** — chunk each doc into ~1000-character pieces (150 overlap) with a
   recursive splitter that breaks on natural boundaries (paragraphs/headers).
   → 100 docs become ~4,100 chunks.
3. **Embed & store** — embed every chunk with a local sentence-transformer
   (`all-MiniLM-L6-v2`, 384-dim) and persist the vectors in a Chroma vector
   store. Done once, offline (`ingest.py`).
4. **Retrieve** — embed the user's question and fetch the top-k most similar
   chunks from Chroma (`query.py`).
5. **Generate (grounded)** — assemble a prompt with the question + retrieved
   chunks + an instruction to answer *only* from them (and to say so when the
   answer isn't in the context), then have Gemini produce a cited answer.
6. **UI** — a Streamlit app (`app.py`) wrapping the above, with cached model
   loading and a form so the LLM only fires on an explicit "Ask".

## Architecture

```
ingest.py   (run once)     query.py  (reusable logic)     app.py  (interface)
────────────────────       ─────────────────────────      ──────────────────
load → split → embed  ──►  chroma_db  ──► retrieve ──►     Streamlit UI
        store vectors                     + grounded
                                          prompt → LLM
```

`query.py` deliberately separates the RAG logic (`build_pipeline`,
`answer_question`) from any interface. `app.py` just imports and calls it —
this is the seam where a **FastAPI backend** would slot in for v2.

## Tech stack (v1)

- **Python**
- **LangChain** (`langchain-core`, `langchain-text-splitters`,
  `langchain-huggingface`, `langchain-chroma`, `langchain-google-genai`)
- **Embeddings:** HuggingFace `sentence-transformers/all-MiniLM-L6-v2`
  (local, free, no API quota)
- **Vector store:** Chroma (persisted to disk in `chroma_db/`)
- **LLM:** Google Gemini `gemini-2.5-flash` (free tier)
- **UI / hosting:** Streamlit + Streamlit Community Cloud

## Setup (run locally)

```bash
# 1. Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Get the Laravel docs (Laravel 13.x) — only needed to rebuild the store
git clone --branch 13.x --depth 1 https://github.com/laravel/docs.git laravel-docs

# 4. Add your Gemini API key (get one at https://aistudio.google.com/apikey)
echo 'GOOGLE_API_KEY=your-key-here' > .env

# 5. Build the vector store (run once)
python ingest.py

# 6. Run the app
streamlit run app.py
```

## Deployment (Streamlit Community Cloud)

The built `chroma_db/` is committed to the repo, so the deployed app loads the
vector store directly — no rebuild, no docs needed on the server. To deploy:

1. Point Streamlit Cloud at this repo, branch `main`, main file `app.py`.
2. Add the API key in **Settings → Secrets** using **TOML** format
   (quotes required):
   ```toml
   GOOGLE_API_KEY = "your-key-here"
   ```
3. A `.streamlit/config.toml` disables the file-watcher in production (see the
   deploy notes below for why).

## What I learned (engineering decisions & tradeoffs)

- **`langchain-community` is archived (2026).** Rather than depend on a sunset
  package, I hand-rolled the doc loader with `pathlib` + `langchain-core` —
  more stable and I understood every line.
- **Local embeddings vs. API embeddings.** I first tried Gemini embeddings to
  avoid shipping PyTorch, but hit two free-tier walls: a 100 requests/min rate
  limit (fixable with throttling) and then a **daily embedding cap** that made
  bulk-indexing ~4,100 chunks impossible on the free tier. I reverted to local
  MiniLM embeddings — no quota, instant build. Lesson: embedding cost/limits
  are a real architectural constraint, not an afterthought.
- **Index-time and query-time embeddings must match.** The store is built and
  queried with the *same* MiniLM model — mixing embedding models would put the
  question and the chunks in different vector spaces.
- **Three deployment bugs worth remembering:**
  - *torchvision `ModuleNotFoundError`.* Streamlit's dev file-watcher walks
    every imported module, which force-loads `transformers`' entire model
    catalog — including vision models that need `torchvision`. Fix: disable the
    watcher in `.streamlit/config.toml` (`fileWatcherType = "none"`); it's not
    needed in production.
  - *API key not found on the cloud.* There's no `.env` on Streamlit Cloud; the
    key lives in `st.secrets`, which doesn't automatically become the
    environment variable LangChain reads. Fix: bridge it in `app.py`.
  - *Secrets format.* The Streamlit Secrets box is TOML — the value needs
    quotes (`KEY = "..."`), unlike a `.env` line.

## Scope

**v1 (this repo):** grounded Q&A over the Laravel docs, single local vector
store, Streamlit UI, deployed to the web.

**Explicitly out of scope for v1:** FastAPI, user accounts, cross-session
memory, agents/tools, multiple documentation sources.

**v2 (planned):** a FastAPI backend so multiple clients can share one
retrieval/LLM service; better retrieval for broad questions (re-ranking,
parent-document / header-aware chunking); additional doc sources; and a proper
evaluation harness.

## Evaluation

Retrieval is evaluated objectively (no human judgment) on a 12-question set
(`eval.py`), each question labelled with two things: the doc file that
canonically answers it (`gold_source`) and a distinctive answer string that
must appear in the retrieved context (`answer_marker`, e.g. `--invokable`).
I measure retrieval at **two levels**:

- **File-level — Hit Rate @ k:** did the correct doc *file* appear in the top-k?
- **Answer-level — Answer-in-Context @ k:** does the concatenated top-k context
  (what actually gets sent to the LLM) *contain the answer text*?

The second metric is the one that matters — the LLM can only answer from what's
in the context — and comparing the two is the interesting result:

| Metric | @3 | @5 | @10 | @15 |
|---|---|---|---|---|
| File-level Hit Rate | 100% | 100% | 100% | 100% |
| **Answer-in-Context** | **67%** | 75% | 83% | **100%** |

**Finding: the coarse metric is misleading.** File-level hit rate is a perfect
100% — but at k=3, only 67% of questions actually have the answer text in
context. The clearest case: for *"how do I create an invokable controller?"*,
`controllers.md` ranks #1 (on its intro chunk), yet the chunk containing the
answer (`--invokable`) sits at rank **15**. A file can rank first while its
answer-bearing chunk is buried far below.

**Tuning result:** raising `k` from 3 → 15 lifts Answer-in-Context from
**67% → 100%**. This is why the app retrieves k=15 — it's the point where every
question's answer is present in context — and the number is now backed by
measurement rather than intuition.

**Tradeoff & root cause:** a high `k` is a brute-force fix — more chunks mean
more tokens, more noise, and slower/costlier calls. The underlying cause is
chunking: a fixed 1000-character split can separate an answer from its section
heading, so the answer chunk embeds poorly against the question. A better fix
than raising `k` is header-aware / larger chunks so the answer ranks high at low
`k` — see v2.

To reproduce: `python eval.py`.

The results for the questions are  : - 

=== File-level Hit Rate @ k (did the right FILE appear?) ===
  @3  : 12/12 = 100%
  @5  : 12/12 = 100%
  @10 : 12/12 = 100%
  @15 : 12/12 = 100%
  @20 : 12/12 = 100%

=== Answer-in-Context @ k (is the ANSWER text in the context?) ===
  @3  : 8/12 = 67%
  @5  : 9/12 = 75%
  @10 : 10/12 = 83%
  @15 : 12/12 = 100%
  @20 : 12/12 = 100%

## Known limitations

- Retrieval returns the *nearest* chunks even for out-of-scope questions; the
  grounding prompt is what prevents hallucinated answers.
- Retrieval relies on a high `k` (15) to guarantee the answer is in context
  (see Evaluation). This works but is brute-force: the root cause is that fixed
  1000-char chunking can split an answer from its section heading. Header-aware
  chunking and/or a stronger embedding model are the real fix — deferred to v2.
- Basic top-k retrieval struggles with broad "explain everything about X"
  questions (a known RAG limitation; deferred to v2).
